# ADR-017 — Перенос подписки при переустановке: entitlement следует за Apple ID (дополняет ADR-013)

- Статус: Accepted
- Дата: 2026-07-23
- Контекст: Apple дважды отклонила «5066 Sonivelle: Music» по Guideline 2.1(a) — «ошибка при
  попытке приобрести подписку» (iPad Air 11" M3, iPadOS 26.5.2, submission
  `e3474a2c-f5a6-4f70-a817-3d97095c3234`, ревью 22.07.2026).
- Связанные: [ADR-013](./ADR-013-storekit-dedup-environment-scoping.md) (environment-scoped
  дедуп, replay-защита), [ADR-005](./ADR-005-coin-wallet-billing.md) (кошелёк, `purchases` +
  `credit_ledger`).

## Контекст

### Хронология инцидента (по логам прода и БД, 2026-07-23)

1. 21.07 04:36 UTC — ревьюер открывает приложение: `POST /v1/auth/guest` → guest `9b0f9bb5`.
2. 04:38 — покупка `yearly_49.99_not_trial` в **Sandbox** (tx `2000001208242767`), verify → ok,
   подписка `active` у `9b0f9bb5`. Полный цикл покупки работает.
3. 04:41 — переустановка приложения (Apple сама рекомендует «удалить предыдущую версию»):
   новый guest `c500a25d`. StoreKit предъявляет чек уже активной подписки → verify отвечает
   `{"status":"rejected","reason":"transaction_already_claimed"}` → на устройстве это выглядит
   как ошибка покупки. Это и есть текст реджекта.
4. 22.07 13:25 — контрольная сессия ревьюера (guest `51188cd7`): только `auth/guest` и
   `billing/balance`; verify до сервера не доходит — sandbox-аккаунт уже «подписан», StoreKit
   падает на устройстве.

### Root cause

Модель «guest-аккаунт на каждую установку» несовместима с глобальным claim транзакции первым
пользователем (ADR-013). После переустановки чек активной подписки навсегда принадлежит
брошенному guest'у: и `purchases/verify`, и `restore` (общий путь
`verify_and_apply_transaction`) отвечают `transaction_already_claimed`. Подписку невозможно
ни купить (StoreKit возвращает существующую транзакцию), ни восстановить. Реджект
воспроизводится у любого реального пользователя, переустановившего приложение.

Replay-защита ADR-013 проектировалась против кражи чужого чека, но не различала «чужой» и
«свой же с прошлой установки»: у guest-модели нет устойчивой идентичности, различить их
серверу нечем — **кроме самого чека**.

## Решение

Владение подписочной цепочкой следует за Apple ID. Валидный подписанный JWS активной подписки
может предъявить только устройство, залогиненное в купивший App Store-аккаунт (подпись
проверяется — ADR-013 D3), поэтому предъявление чека — достаточное доказательство владения.
Индустриальный стандарт (transfer behavior в RevenueCat/Adapty) делает то же самое.

`BillingService._apply`, случай «ключ занят другим пользователем»:

- **Подписка** (`kind=subscription`, environment `Production`/`Sandbox`) — перенос
  (`_transfer_subscription_chain`):
  1. `purchases.reassign_original_transaction` — все строки цепочки
     (`original_transaction_id`, в рамках того же environment) переезжают на текущего
     пользователя → webhooks (`find_user_by_original_transaction`) находят актуального
     владельца, renewal/refund применяются к нему;
  2. `credits.expire_foreign_subscriptions` — `active`-подписки других пользователей на эту
     цепочку гасятся (`expired`): один Apple ID — один активный владелец;
  3. `upsert_subscription` текущему пользователю → `active`.
  Ответ: `{"status":"ok","deduplicated":true,"reason":"subscription_transferred"}`.
- **Монеты не переначисляются**: ledger-ключ `purchase:{dedup_key}` уже погашен прежним
  владельцем. Иначе цикл «reinstall → новый guest → re-verify» был бы фармом грантов.
- **Коин-пак** — по-прежнему `rejected/transaction_already_claimed`: переносить нечего
  (consumable уже потреблён прежним аккаунтом), а новый пак — это всегда новый `transactionId`.
- **Xcode** — в переносе не участвует: ID не уникальны между пользователями (счётчики
  `0,1,2…`), перенос по `original_transaction_id` отдал бы чужие тестовые покупки. Дедуп
  Xcode и так per-user (ADR-013), claim-конфликт там невозможен.

Тот же `_transfer_subscription_chain` вызывается и при **вставленной** новой транзакции
подписки (re-subscribe/renewal той же цепочки после переустановки): история переезжает,
прежний владелец гасится, монеты за новый период начисляются штатно (`newly=true`).
Для первой покупки оба апдейта — no-op.

## Альтернативы (отклонены)

- **Отвечать `ok` без переноса** — подписка «покупается», но entitlement остаётся у брошенного
  аккаунта: пользователь платит и не получает контент. Хуже честного отказа.
- **Сузить дедуп до `(user_id, tx)`** — уже отклонено в ADR-013: ломает replay-защиту паков.
- **Переносить и кошелёк монет старого guest'а** — привлекательно (монеты брошенного аккаунта
  недостижимы), но происхождение монет смешанное (другие паки, рефанды), а чек доказывает
  владение только подписочной цепочкой. Отложено (см. Последствия).
- **Требовать Sign in with Apple до покупки** — решает идентичность, но добавляет фрикцию на
  paywall и требует клиентского релиза; не отменяет необходимость серверного переноса для уже
  существующих guest-покупок.

## Последствия

- Сценарий Apple-ревью (покупка → переустановка → покупка/restore) проходит: verify и restore
  возвращают `ok` и активируют подписку текущему пользователю.
- Устройство прежней установки при следующем verify того же чека получит перенос обратно —
  «последний предъявивший владеет». Для одного физического пользователя это корректно;
  канонично решается через `appAccountToken` / Sign in with Apple (вне scope).
- `credit_ledger` исторически хранит грант за прежним владельцем, `purchases` — за актуальным:
  расхождение атрибуции задокументировано здесь; идемпотентность гранта не нарушена (ключ
  глобален).
- Монеты периода, начисленные брошенному аккаунту, не переезжают — известное ограничение
  (TD: рассмотреть перенос кошелька guest→guest при переносе подписки).
- Открытые prod-риски вне этого ADR (зафиксированы при расследовании):
  `APPLE_STOREKIT_TRUST_XCODE_TEST_CERTS=true` на проде (вектор бесплатных монет) и пустые
  ключи App Store Server API (нет серверной сверки статуса подписки).
