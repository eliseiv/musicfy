# ADR-003 — Health gate перед `docker image prune`

- Статус: Accepted
- Дата: 2026-06-19
- Контекст: CI/CD musicfy (`.github/workflows/deploy.yml`), rollback-стратегия

## Context

Деплой пересобирает образ `api` (`up -d --build`). После пересборки на хосте накапливаются
dangling-образы, их чистят `docker image prune -f`. Но образ предыдущей (рабочей) версии —
единственный артефакт для быстрого отката: если новый деплой нездоров, нужно сохранить
возможность вернуться. Безусловный `prune` сразу после `up` мог бы удалить рабочий образ
ещё до того, как подтверждено, что новая версия исправна.

## Decision

После `docker compose up -d --build` выполняется **health gate**: до ~120 с (24 × 5 с)
ожидается `State.Health.Status == healthy` у `musicfy-api-1` (healthcheck — `GET /healthz`).

- Если api не стал healthy в таймаут — деплой **падает** (`exit 1`), `prune` **не выполняется**,
  предыдущий образ остаётся для отката.
- `docker image prune -f` выполняется **только после** подтверждения health.

## Consequences

- (+) Проваленный деплой не уничтожает рабочий образ — откат остаётся возможным.
- (+) Зелёный деплой = подтверждённо живой `/healthz`, а не просто «контейнер поднялся».
- (+) CI явно сигнализирует о провале (красный workflow), а не «успешно, но сломано».
- (−) Деплой может длиться до ~2 мин дольше при проблемах со стартом.
- (−) Это защита уровня образа, не полноценный blue/green; откат — повторный деплой
  предыдущего коммита (DEPLOYMENT.md §5).

## Alternatives

- **Безусловный `prune` сразу после `up`** — отклонено: удаляет образ для отката.
- **Без health gate (деплой = успешен по факту `up`)** — отклонено: маскирует сломанный старт
  (упавшие миграции, битый конфиг) как успешный деплой.
