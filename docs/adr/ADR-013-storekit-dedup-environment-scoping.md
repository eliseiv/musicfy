# ADR-013 — Дедуп StoreKit-покупок: environment-scoped ключ + trust anchor вместо глобального bypass

- Статус: Accepted
- Дата: 2026-07-13
- Контекст: на проде (`zavionix.shop`) коин-паки не начисляются при покупке из Xcode StoreKit
  (local testing). Совершенно новый гость с балансом 0 шлёт `POST /v1/billing/purchases/verify`
  с `transactionId="0"` → ответ `{"status":"ok","deduplicated":true}`, баланс остаётся 0.
  Первая покупка пользователя молча не начисляется.
- Связанные: [ADR-005](./ADR-005-coin-wallet-billing.md) (единый кошелёк монет, `purchases` +
  `credit_ledger`), [ADR-010](./ADR-010-lyrics-sync-billing.md) (идемпотентность списаний по ключу),
  [ADR-001](./ADR-001-fail-fast-db-password.md) (прецедент fail-fast на небезопасный prod-конфиг).

## Контекст

### Симптом

Воспроизведён чёрным ящиком на проде: новый пользователь, `com.musicfy.coins.small` (100 монет),
`transactionId="0"` → `HTTP 200 {"status":"ok","deduplicated":true}`, баланс 0 → **0**. Контроль:
тот же продукт с уникальным `transactionId` → баланс 0 → 100. Каталог продуктов в порядке,
`APPLE_STOREKIT_VERIFY_SIGNATURE=false` на проде активна (подписка грантится).

### Root cause: дедуп глобален по `transaction_id`, в двух независимых слоях

1. **Слой `purchases`** — `models/billing.py:179`
   `UniqueConstraint("transaction_id", name="uq_purchases_transaction_id")`; вставка через
   `repositories/products.py:50` `.on_conflict_do_nothing(constraint="uq_purchases_transaction_id")`.
   Уникальность — по одному `transaction_id`, **без `user_id` и без окружения**.
2. **Слой `credit_ledger`** — `services/billing_service.py:141`
   `idempotency_key=f"purchase:{transaction_id}"` при `UniqueConstraint("idempotency_key")`
   (`models/billing.py:144`). Тоже глобальный.

`_apply` (`billing_service.py:62-90`) вызывает `_grant_coins` **независимо от `newly`** — это
маскировало расхождение слоёв: даже если строка `purchases` не вставилась, грант «пытается»
пройти, но глушится вторым (тоже глобальным) ключом в леджере. Итог — `deduplicated: true` и
тишина.

Почему ломается именно тест-режим: в Xcode StoreKit local testing `transactionId` — мелкие
счётчики (`0`, `1`, `2`, …), которые **обнуляются** при *Delete All Transactions* и сталкиваются
с уже засчитанными в проде записями — в том числе с прошлыми покупками того же разработчика.
Сервер честно считает это повтором.

### Конфликт требований

- **PROD:** настоящий Apple `transactionId` глобально уникален. Глобальный дедуп — это
  **защита от replay**: пользователь B не должен погасить чек пользователя A. Терять нельзя.
- **TEST:** ID переиспользуются и сталкиваются даже в рамках одного пользователя. Нужна
  возможность повторно покупать.

Сужение `UNIQUE` до `(user_id, transaction_id)` — очевидный, но **неверный** фикс: оно
(а) ломает cross-user replay-защиту в бою (B применяет чек A под своим `user_id` — ключ свободен)
и (б) **не решает** исходную проблему, т.к. коллизия у разработчика — с его же собственными
прошлыми записями.

### Сопутствующая находка (P0, security)

`APPLE_STOREKIT_VERIFY_SIGNATURE=false` **на проде** означает, что `_verify_and_decode`
(`providers/billing/apple.py:77-82`) декодирует JWS **без проверки подписи**. Любой
аутентифицированный пользователь может отправить самодельный неподписанный JWT
`{"transactionId": "<любой>", "productId": "com.musicfy.coins.large"}` и **бесплатно намайнить
монеты**. Это не побочная деталь бага — это активная дыра в бою, и именно она вынудила
рассматривать «режим верификации» как дискриминатор. Решение ниже устраняет и её.

## Решение

Три связанных решения. Они неразделимы: доверять `environment` как дискриминатору дедупа можно
**только** если payload криптографически верифицирован (D3), поэтому D1 без D3 недопустим.

### D1. Дедуп-ключ вычисляется из окружения транзакции, а не из конфига сервера

Вводится единственная функция-источник истины — `BillingService._dedup_key(user_id, tx) -> str`.
`environment` берётся из **верифицированного** payload (StoreKit 2 claim `environment`:
`Production` | `Sandbox` | `Xcode`), а не из настроек сервера.

| environment | Дедуп-ключ | Область | Обоснование |
|---|---|---|---|
| `Production` | `Production:{transaction_id}` | **глобально** (все пользователи) | Apple гарантирует глобальную уникальность `transactionId`. Replay-защита сохранена в полном объёме: чек A, применённый один раз, не применится ни у B, ни повторно у A. |
| `Sandbox` | `Sandbox:{transaction_id}` | **глобально** | Тот же контракт уникальности внутри Sandbox. Namespace отделяет Sandbox от Production — закрывает латентную коллизию, при которой sandbox-ID мог «затенить» боевой ID (сегодня они в одном пространстве). |
| `Xcode` | `Xcode:{user_id}:{transaction_id}:{purchase_date_ms}` | **на пользователя + на момент покупки** | Гарантии уникальности нет (счётчик, обнуляемый при *Delete All Transactions*). `purchase_date_ms` (claim `purchaseDate`) даёт энтропию: повторная покупка после сброса — это **новый** `purchaseDate` → новый ключ → монеты начисляются. Повтор **того же** payload (тот же `purchaseDate`) → тот же ключ → дедуп. Идемпотентность по полному payload сохранена. |

Ключевое свойство: **минимальность боевого ключа**. Для `Production` в ключ входит только
`transaction_id` — самая узкая возможная идентичность, поэтому её нельзя «размыть» ничем из
payload. Именно поэтому вариант «добавить `purchase_date` в ключ для всех окружений» отклонён
(см. Альтернативы) — он ослабляет боевой инвариант ради тест-режима.

`user_id` в `Xcode`-ключе не является защитой (Xcode-ветка и так закрыта пином сертификата, D3) —
он лишь исключает любое кросс-аккаунтное влияние тестовых ID.

### D2. Оба слоя используют один и тот же ключ

Расхождение слоёв — половина этого бага. Контракт:

- `purchases`: новая колонка `dedup_key` (`VARCHAR(255)`, `NOT NULL`), `UNIQUE
  (dedup_key)` = `uq_purchases_dedup_key`. Вставка — `on_conflict_do_nothing(constraint=
  "uq_purchases_dedup_key")`. Старый `uq_purchases_transaction_id` **снимается** (заменяется
  неуникальным индексом `ix_purchases_transaction_id` — по нему ходят restore/саппорт).
- `credit_ledger`: `idempotency_key = f"purchase:{dedup_key}"`. Схема не меняется
  (`uq_credit_ledger_idempotency_key` уже есть, `String(160)` вмещает
  `purchase:Production:{19-значный id}`).
- `_apply` начисляет монеты **только при `newly is True`** (строка `purchases` реально
  вставлена). Оба insert'а идут в одной транзакции (`session.begin()`), поэтому частичного
  состояния «покупка есть, гранта нет» не бывает. Внутренняя идемпотентность `_grant_coins`
  по ledger-ключу остаётся как defence-in-depth, но перестаёт быть единственным барьером.

`purchases` дополнительно сохраняет `environment` и `purchase_date` — для аудита и для того,
чтобы тестовые начисления на проде были отделимы от боевых (фильтр/выгрузка/зачистка).

Подписки не затрагиваются: у каждого продления — новый `transactionId` → новый ключ → новый
грант за период; `_update_subscription_state` остаётся идемпотентным upsert'ом.

### D3. Trust anchor выбирается по цепочке x5c; глобального bypass в проде нет

`APPLE_STOREKIT_VERIFY_SIGNATURE=false` перестаёт быть «выключателем безопасности в бою»:

- **Fail-fast при старте** (прецедент ADR-001): `APP_ENV=prod` + `APPLE_STOREKIT_VERIFY_SIGNATURE
  =false` → приложение **не поднимается**. Флаг остаётся легальным только для `APP_ENV` ∈
  `{dev, test}` (юнит-тесты с синтетическими токенами — там `environment` берётся из claim,
  дефолт `Xcode`).
- **Подпись проверяется всегда**, но корень цепочки может быть одним из двух:
  - корень == закреплённый **Apple Root CA - G3** → payload подлинный, claim `environment`
    (`Production` / `Sandbox`) можно доверять;
  - корень ∈ **`APPLE_STOREKIT_TEST_ROOT_CERTS`** (новый конфиг: список PEM корневых
    сертификатов StoreKit Test, экспортируемых из Xcode конкретного разработчика) → `environment`
    принудительно = `Xcode`, независимо от claim;
  - иначе → `WebhookPayloadInvalid` (`untrusted_root`), как сейчас.
- Дефолт `APPLE_STOREKIT_TEST_ROOT_CERTS = []` → на проде Xcode-ветка **выключена**. Её
  включение — явный, аудируемый акт (пин сертификата конкретной машины), а не «тумблер
  доверия всему подряд». См. Q-BILL-1.

Так `environment` становится **криптографически удостоверенным** свойством payload'а, а не
самоаттестацией клиента — только на этом и держится D1.

## Влияние на безопасность в бою

| Вектор | До | После |
|---|---|---|
| Подделка неподписанного JWS → бесплатные монеты | **Открыт** (verify=false на проде) | Закрыт: fail-fast не даст поднять prod с bypass; подпись проверяется всегда. |
| Replay чужого боевого чека (B применяет чек A) | Закрыт (глобальный `transaction_id`) | **Закрыт так же**: `Production:{tx}` глобален. Инвариант не ослаблен. |
| Повторное применение своего боевого чека (restore-спам) | Закрыт | **Закрыт так же.** |
| Sandbox-ID затеняет боевой ID | Открыт (одно пространство ключей) | Закрыт: раздельные namespace. |
| Клиент объявляет себя `Xcode`, чтобы обойти дедуп | н/д | Закрыт: `environment` берётся из верифицированного payload; ветка `Xcode` требует пина корневого сертификата, пустого по умолчанию. |
| Тестовые монеты на проде | Неотличимы | Помечены (`purchases.environment != 'Production'`), отделимы, зачищаемы. |

Итог: ни один боевой инвариант не ослаблен; два ранее открытых вектора закрыты.

## Миграция

`0016_purchase_dedup_key` (`down_revision = "0015_soft_delete"`; текущая голова цепочки — `0015`).

**upgrade:**
1. `purchases` += `environment VARCHAR(16) NOT NULL DEFAULT 'Production'` — существующие строки
   применялись как боевые, бэкфилл дефолтом корректен.
2. `purchases` += `purchase_date TIMESTAMPTZ NULL` (аудит + материал ключа для `Xcode`).
3. `purchases` += `dedup_key VARCHAR(255) NULL`; бэкфилл
   `UPDATE purchases SET dedup_key = 'Production:' || transaction_id`; затем `SET NOT NULL`.
4. `CREATE UNIQUE INDEX/CONSTRAINT uq_purchases_dedup_key ON purchases (dedup_key)`.
5. `DROP CONSTRAINT uq_purchases_transaction_id`; `CREATE INDEX ix_purchases_transaction_id`
   (неуникальный).
6. **Обязательный бэкфилл леджера** (иначе — двойное начисление после деплоя):
   `UPDATE credit_ledger SET idempotency_key = 'purchase:Production:' || ref_id
    WHERE ref_type = 'transaction' AND idempotency_key LIKE 'purchase:%'`.
   Без этого шага уже применённый боевой чек имеет старый ключ `purchase:{tx}`, а новый код
   ищет `purchase:Production:{tx}` — restore после деплоя начислил бы монеты повторно.
   Шаг 6 и шаги 1-5 обязаны быть в одной миграции (одна транзакция).

**downgrade:** обратный бэкфилл ключей леджера (`'purchase:' || ref_id`), drop
`uq_purchases_dedup_key` / `ix_purchases_transaction_id`, drop колонок, восстановление
`uq_purchases_transaction_id`. Восстановление UNIQUE может упасть при наличии строк с
одинаковым `transaction_id` в разных окружениях — это ожидаемо и допустимо (downgrade
best-effort, ср. [TD-001](../100-known-tech-debt.md#td-001)).

## Разблокировка разработчика

Его ID (`0`, `1`, `2`, …) уже «сожжены» в проде: лежат в `purchases` с
`uq_purchases_transaction_id` и в `credit_ledger` с ключами `purchase:0`, `purchase:1`, …

**Немедленный стоп-гэп (до выката фикса) — обязательная часть, а не опция:**

1. **`APPLE_STOREKIT_VERIFY_SIGNATURE=true` на проде — сейчас.** Пока флаг `false`, прод
   раздаёт монеты любому, кто умеет собрать JWT. Это приоритетнее самого бага.
2. Разработчик тестирует StoreKit против **staging/dev-окружения** (`APP_ENV=dev`,
   `APPLE_STOREKIT_VERIFY_SIGNATURE=false`, отдельная БД) — там коллизий с боевыми записями нет
   и bypass легален.
3. Если баланс нужен именно на его **проде**-аккаунте прямо сейчас — разовый ручной грант в
   кошелёк с ledger-записью `kind=credit_purchase`, `reason="manual_test_grant"`,
   `idempotency_key="manual:{uuid4}"`. Это **не покупка**: `purchases` не трогается, аудит
   остаётся честным.

**Штатный путь (после выката `0016` + D1/D3):** ручная зачистка **не требуется**. Старые записи
живут с ключами `Production:{tx}` / `purchase:Production:{tx}`, новые Xcode-покупки получают
ключи `Xcode:{user_id}:{tx}:{purchase_date_ms}` — пространства ключей не пересекаются
**по построению**, «сожжённые» ID автоматически перестают быть сожжёнными. Требуется лишь
пин его StoreKit Test root-сертификата в `APPLE_STOREKIT_TEST_ROOT_CERTS` (см. Q-BILL-1).

`DELETE` его строк из `purchases` / `credit_ledger` — **отклонено** как практика: уничтожает
аудит-трейл и после этого фикса не нужен.

## Альтернативы (отклонены)

- **`UNIQUE (user_id, transaction_id)`.** Ломает cross-user replay-защиту в бою (B применяет чек
  A) и не решает исходную коллизию (у разработчика конфликт с собственными прошлыми записями).
  Отклонено по обоим пунктам.
- **`purchase_date` в ключе для всех окружений.** Работает, но размывает минимальную боевую
  идентичность: любой сценарий, где Apple предъявит тот же `transactionId` с иным `purchaseDate`,
  превращается в повторное начисление. Ради тест-режима платить ослаблением боевого инварианта
  не нужно — окружения разделимы (D1).
- **Отключать дедуп при `verify_signature=false`.** Дедуп-режим оказался бы завязан на
  server-wide флаг: одна ошибка конфигурации (ровно текущая!) снимала бы одновременно и проверку
  подписи, и replay-защиту — отказы складываются. `environment` из верифицированного payload —
  свойство самой транзакции и потому надёжный дискриминатор; флаг конфигурации — нет.
- **Специальный `user.is_test` + отдельный дедуп для тест-аккаунтов.** Дискриминатор берётся из
  нашей БД, а не из транзакции; тестовый аккаунт с боевым чеком получил бы ослабленный дедуп.
  Сложнее и слабее D1.

## Последствия

**Плюсы:** тест-режим работает (повторные покупки после сброса Xcode начисляются); боевая
replay-защита сохранена побайтово; закрыта дыра с неподписанными JWS; закрыта латентная
Sandbox/Production-коллизия; два слоя дедупа сведены к одному источнику истины; тестовые
начисления на проде отделимы от боевых.

**Минусы / цена:** миграция с обязательным бэкфиллом `credit_ledger` (пропуск шага 6 = двойные
начисления — риск, требующий внимания при выкате); `APPLE_STOREKIT_TEST_ROOT_CERTS` нужно
обновлять при смене машины/Xcode разработчика (ops-нагрузка, [TD-010](../100-known-tech-debt.md#td-010));
downgrade миграции best-effort.

**Обязательства:** `APPLE_STOREKIT_VERIFY_SIGNATURE=true` на проде — инвариант, защищённый
fail-fast'ом; ключ дедупа изменяется только новым ADR (изменение формата ключа требует бэкфилла
леджера).
