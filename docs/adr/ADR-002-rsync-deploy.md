# ADR-002 — Rsync-деплой на сервер вместо git-pull

- Статус: Accepted
- Дата: 2026-06-19
- Контекст: CI/CD musicfy (`.github/workflows/deploy.yml`)

## Context

Сервер `87.239.135.154` — общий, и репозиторий musicfy приватный. Класссический паттерн
«сервер делает `git pull`» требует положить на сервер креды/deploy-key с доступом к приватному
GitHub-репозиторию. Это расширяет поверхность атаки (общий сервер получает доступ к исходникам
в GitHub) и усложняет ротацию. Нужен способ доставить код без выдачи серверу доступа в repo.

## Decision

Деплой выполняется по `push` в `main` через GitHub Actions, который **rsync'ит рабочее дерево**
runner'а на сервер и затем по SSH запускает сборку:

- `rsync -az --delete` в `/opt/musicfy/` с исключением `.git`, `.env`, `.venv`, кешей;
- SSH-доступ — по ключу из `SSH_PRIVATE_KEY`, host проверяется через `SSH_KNOWN_HOSTS`
  (`StrictHostKeyChecking=yes`);
- на сервере — `docker compose -f docker-compose.prod.yml up -d --build`.

Сервер не хранит GitHub-кредов и не ходит в репозиторий. Секреты CI — в GitHub Secrets
(`SSH_HOST`, `SSH_USER`, `SSH_PRIVATE_KEY`, `SSH_KNOWN_HOSTS`).

## Consequences

- (+) Общий сервер не получает доступ к приватному GitHub-репозиторию.
- (+) Деплоится ровно то дерево, что собрал runner; нет состояния git на сервере.
- (+) `--exclude='.env'` гарантирует, что серверные секреты не перезаписываются.
- (−) `--delete` приводит `/opt/musicfy/` к состоянию репозитория: хранить там посторонние
  серверные файлы (кроме `.env`) нельзя — они будут удалены. Зафиксировано в DEPLOYMENT.md §6.
- (−) Нужна корректная запись `SSH_KNOWN_HOSTS` (иначе деплой падает на проверке host-ключа).

## Alternatives

- **`git pull` на сервере** — отклонено: требует GitHub-креды на общем сервере.
- **Container registry (build в CI, pull образа)** — отклонено сейчас: требует приватный
  registry и креды на сервере; избыточно для текущего масштаба. Возможный будущий путь при
  росте требований.
