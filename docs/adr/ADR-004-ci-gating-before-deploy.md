# ADR-004 — CI gating перед деплоем

- Статус: Accepted
- Дата: 2026-06-19
- Контекст: CI/CD musicfy (`.github/workflows/deploy.yml`), дополняет ADR-002

## Context

Ранее `deploy.yml` триггерился напрямую на `push` в `main` (см. ADR-002), независимо от
результата CI. Это допускало деплой коммита, который не прошёл (или ещё не прошёл) проверки
CI: гонка между `push`-триггером деплоя и прогоном CI могла выкатить непротестированный код.
Кроме того, при прямом `push`-триггере на сервер уезжало рабочее дерево runner'а на момент
запуска job, что при быстрых последовательных push'ах не гарантировало соответствие
задеплоенного состояния тому коммиту, который реально проверялся.

Нужно: деплоить только то, что прошло CI зелёным, и ровно тот коммит, на котором CI стал
зелёным. При этом механика доставки (rsync рабочего дерева, ADR-002) остаётся прежней.

## Decision

Деплой запускается **по завершении workflow «CI»** (`on: workflow_run: workflows: ["CI"],
types: [completed]`), а не напрямую по `push`. Job-level условие `if` пропускает деплой только
при одновременном выполнении:

- `github.event.workflow_run.conclusion == 'success'` — CI завершился успешно;
- `github.event.workflow_run.head_branch == 'main'` — CI был на ветке `main`;
- `github.event.workflow_run.event == 'push'` — CI был запущен `push`'ем (PR-прогоны CI
  деплой не вызывают).

`checkout` выполняется с `ref: ${{ github.event.workflow_run.head_sha }}` — деплоится именно
протестированный коммит (SHA, на котором CI стал зелёным), а не последний `main` на момент
запуска деплоя.

Concurrency-группа — `deploy-${{ github.workflow }}` (с сохранённым `cancel-in-progress:
false`). Группировка по `github.workflow`, а не по `github.ref`, потому что при триггере
`workflow_run` `github.ref` указывает на default branch и как ключ группировки нестабилен.

**Связь с ADR-002.** rsync-механизм доставки и запуска (rsync рабочего дерева на сервер →
`docker compose up -d --build` по SSH, сервер без доступа к репозиторию) этим решением **не
меняется**. ADR-004 **дополняет** ADR-002, изменяя только триггер и gating деплоя, а не способ
доставки. ADR-002 остаётся Accepted.

## Consequences

- (+) Деплоится только код, прошедший CI зелёным, — исключён выкат непротестированного
  коммита.
- (+) Деплоится ровно протестированный SHA (`workflow_run.head_sha`), без гонок с
  последующими push'ами в `main`.
- (+) PR-прогоны CI не приводят к деплою (gate `event == 'push'` + `head_branch == 'main'`).
- (+) Concurrency по `github.workflow` стабильна при триггере `workflow_run`.
- (−) Деплой стартует не мгновенно после `push`, а только после завершения CI — общее время
  доставки до прода увеличивается на длительность CI.
- (−) Появляется неявная связь между именами: workflow деплоя завязан на имя workflow «CI»
  (строка `workflows: ["CI"]`) — переименование CI-workflow молча отключит деплой.

## Alternatives

- **Прямой `push`-триггер деплоя (как было, ADR-002)** — отклонено: допускает деплой
  непротестированного коммита и гонку с CI.
- **`needs:`-зависимость деплой-job от CI-job в одном workflow** — отклонено: CI и deploy
  остаются отдельными workflow по организационным причинам; `workflow_run` связывает их без
  слияния в один файл.
- **Группировка concurrency по `github.ref`** — отклонено: при `workflow_run` `github.ref`
  указывает на default branch и не является стабильным ключом для сериализации деплоев.
